Automatic Accountant: Ultimate Setup Guide

Created by Joseph Tay | Licensed under the MIT License.

This comprehensive guide serves as a manual for setting up the Automatic Accountant pipeline from scratch. When transitioning from a dummy test account to actual business accounts, follow these sequential steps to establish the required Cloud APIs, connect the environment variables, and deploy the infrastructure to AWS.

🛠️ Prerequisites

Before proceeding, ensure the following tools are present in your development environment:

  1. Python 3.9+: The core runtime language.
  2. Docker: Containerizes the application for identical local and cloud execution.
  3. Terraform: Infrastructure as Code (IaC) tool utilized to automatically provision cloud servers by reading the main.tf configuration.
  4. AWS CLI: Enables secure terminal communication with Amazon Web Services.
  5. Git/VSCode: For modifying source code and environment variable (.env) configurations.

📌 Phase 1: Google Cloud Platform (GCP) Setup

To grant the application required access to Google Calendar and Google Sheets without exposing a personal password, a secure "Service Account" (Robot User) must be created within the Google Cloud Platform.

Step 1.1: Project Initialization

  • Navigate to the Google Cloud Console.
  • Select the Project Dropdown (top left) and click New Project.
  • Assign the name Automatic Accountant and select Create.

Step 1.2: API Enablement

  • In the sidebar, select APIs & Services > Library.
  • Search for and Enable both the Google Calendar API and the Google Sheets API.

Step 1.3: Service Account Generation

  • Navigate to APIs & Services > Credentials.
  • Select + CREATE CREDENTIALS > Service account.
  • Name the account "Billing Robot" and assign the Basic > Editor role.
  • Copy the generated Service Account email address.

Step 1.4: Credential Extraction

  • Under the Service Account details, navigate to the KEYS tab.
  • Select ADD KEY > Create new key > JSON.
  • Relocate the downloaded file into the automatic-accountant project directory and rename it precisely to credentials.json.

📌 Phase 2: Integration Configuration

Step 2.1: Authorization

Google enforces strict privacy defaults. The service account requires explicit invitations to access the necessary files.

  • In Google Calendar, navigate to "Settings and sharing", add the service account email, and grant "See all event details" permission.
  • In Google Sheets, click the "Share" button and invite the service account email as an "Editor".

Step 2.2: Identifying Unique IDs

  • Calendar ID: Located in Calendar "Settings". Typically formatted as c_xyz...@group.calendar.google.com.
  • Spreadsheet ID: Extracted from the Sheet's URL (the string directly following /d/).

Step 2.3: Data Structure Formatting

The target Google Sheet must contain the following exact headers in Row 1 (A1 to G1) to parse incoming matrix data correctly:

  • Event Date | Client | Service | Amount Owed | Status | Payment Received Date | Comments (Manual Entry)

Step 2.4: Event Syntax Standards

When scheduling, the logic evaluates the Event Title utilizing strict formatting rules:

Client Name [$Rate] - Service Type

  • Dynamic Rates: Encapsulate precise rates within brackets or parentheses (e.g., [$250]). If omitted, the system falls back to the .env HOURLY_RATE.
  • Service Type: Delimit with a hyphen (e.g., - Consulting).
  • Time Bounds: "All-Day" events are incompatible; explicit start and end times are required for duration-based calculations.

📌 Phase 3: Amazon Web Services (AWS) Provisioning

AWS serves as the execution layer. To maintain security, an Identity and Access Management (IAM) User must be provisioned following strict "Least Privilege" protocols.

Step 3.1 & 3.2: Terraform Deployment Identity

Create an IAM User specifically for Terraform execution (e.g., terraform-admin), providing CLI access exclusively. Attach a custom JSON policy that singularly permits Lambda, EventBridge, ECR, and local IAM Role creation, explicitly preventing account-wide escalation commands.

📌 Phase 4 & 5: Environment Variables & Deployment

Construct the .env file within the root directory utilizing the compiled IDs and Keys:

GOOGLE_SHEETS_SPREADSHEET_ID=1BxiMVs0X_xxxxxxxxxxxxxxxxxxx
AWS_ACCESS_KEY_ID=AKIA...
AWS_SECRET_ACCESS_KEY=skJhg...
HOURLY_RATE=150
CALENDAR_ID=c_xyz123@group.calendar.google.com
SYNC_DAYS_BACK=1

Final Execution

1. Validate logic locally utilizing ./test_locally.sh.

2. Provision the AWS infrastructure using Terraform:

terraform init
terraform plan
terraform apply

3. Authenticate Docker with AWS, build the image (enforcing AMD64 architecture for cloud compatibility), and push the container to the newly provisioned Elastic Container Registry (ECR).

The system is now fully automated via AWS EventBridge.

📌 Phase 6: Infrastructure Teardown

To systematically eradicate the architecture and prevent any potential Free Tier billing overages:

  1. Navigate to the AWS Console (ECR) and manually delete all hosted Docker images (Terraform will abort destruction if images remain).
  2. Execute terraform destroy within the project directory to dismantle all connected CloudWatch rules, IAM profiles, and Lambda containers.
