The Automatic Accountant

I was spending too much time manually migrating my consulting appointments from Google Calendar to Google Sheets. I originally planned to build this project on a Raspberry PI, but hardware prices have increased quite a bit. Plus, I wanted more hands-on experience working with Cloud projects (specifically learning how to work with Terraform and Docker). So, I decided to try building this pipeline using AWS, Lambda, Docker, Terraform, and Antigravity.

Problems Faced & the Best Practices

During the development of this serverless pipeline, I encountered several engineering challenges that required strict adherence to AWS Cloud Best Practices to overcome completely:

1. Authorization & Identity Management

The Challenge: Establishing a secure connection between my AWS automation and Google APIs without hardcoding sensitive OAuth tokens directly into the application code.

The Solution: I adopted the Principle of Least Privilege by isolating all Google authentication tokens within the AWS Systems Manager Parameter Store. The Lambda execution role was explicitly granted ssm:GetParameter access only to the exact securely encrypted tokens it needed, ensuring zero credential leakage.

2. Deployment Size Limitations

The Challenge: Standard AWS Lambda environments have a deployment package size limit (50MB zipped), but the required Google API Python libraries exceeded this when bundled traditionally.

The Solution: Rather than forcing a massive zip file, I re-architected the deployment strategy to use Docker Container Images via AWS ECR. By packaging the runtime and all dependencies into a streamlined Docker container, I circumvented the layer limits completely while also ensuring the environment remained identical between my local WSL machine and the cloud.

3. Cold Start & Zero-Cost Constraints

The Challenge: Guaranteeing the pipeline stayed entirely within the Free Tier limits meant avoiding any always-on compute instances (like EC2 or RDS) that might accumulate passive billing charges.

The Solution: I transitioned strictly to a Serverless Event-Driven architecture using AWS EventBridge. By triggering the execution via a serverless cron job, compute power is only spun up exactly when my calendar needs synchronizing once a day, resulting in absolute zero passive computational cost.

Infrastructure as Code Documentation

To ensure this deployment is completely reproducible with no manual clicks, I relied heavily on Terraform to provision the cloud architecture. Terraform code is written in HashiCorp Configuration Language (HCL), which is highly declarative and easy to read.

For instance, to trigger the AWS Lambda automatically every single day, an AWS EventBridge cron job is used. The proper way to code this connection requires separating the timing schedule from the target mapping. The aws_cloudwatch_event_rule resource dictates the execution time, while the aws_cloudwatch_event_target bonds that schedule to our specific Lambda ARN. Finally, a strict Identity and Access Management (IAM) aws_lambda_permission block is required to securely validate EventBridge's authority to actually invoke the function. 

# --- EventBridge Target for Ledger Sync ---
resource "aws_cloudwatch_event_rule" "daily_ledger_sync" {
  name                = "daily_ledger_sync_rule"
  description         = "Trigger Ledger Sync Lambda daily"
  schedule_expression = "rate(1 day)"
}

resource "aws_cloudwatch_event_target" "ledger_sync_target" {
  rule      = aws_cloudwatch_event_rule.daily_ledger_sync.name
  target_id = "LedgerSyncLambdaTarget"
  arn       = aws_lambda_function.ledger_sync.arn
}

resource "aws_lambda_permission" "allow_eventbridge" {
  statement_id  = "AllowExecutionFromEventBridge"
  action        = "lambda:InvokeFunction"
  function_name = aws_lambda_function.ledger_sync.function_name
  principal     = "events.amazonaws.com"
  source_arn    = aws_cloudwatch_event_rule.daily_ledger_sync.arn
}
